Clinic Front Desk System — Product & Engineering Plan

You will build this with React + MUI (frontend), Node.js + Express (backend), and MongoDB (database). The PDF originally specified NestJS/MySQL/Next.js/Tailwind; your stack is a valid equivalent—Express and MongoDB map cleanly to the same features.

1. Requirements Dissection (What You're Building)

Break the PDF into four pillars so it feels less overwhelming:

PillarWhat it meansYour responsibility
AuthFront desk staff log in; no public signupLogin page, JWT, protected routes
DoctorsCRUD for doctor profiles (specialization, gender, location, availability)Doctor list/add/edit/delete + search/filter
QueueWalk-ins get a queue number; status = Waiting → With Doctor → CompletedQueue list, add patient, update status
AppointmentsBook, view, reschedule, cancel; status = Booked / Completed / CanceledSlots view, book/reschedule/cancel, status updates

Optional (later): Deploy FE + BE, queue prioritization (e.g. "urgent").

You do not need to build patient self-service or doctor login for the core scope—only front desk staff.

2. High-Level Architecture

Frontend (React MUI): Pages, Components, API Services, State
        ↓ REST API
Backend (Express): Routes, Controllers, Auth Middleware, Models
        ↓ Mongoose
MongoDB: Collections
  • Frontend: React app (e.g. Vite or CRA) with MUI, calls Express REST API.
  • Backend: Express server, JWT auth, Mongoose for MongoDB.
  • Deploy: FE and BE as two separate deployables (e.g. Vercel/Netlify for FE, Railway/Render/Fly.io for BE).

3. Where to Create the Project

Create one parent folder (e.g. clinic-front-desk) and two projects inside it:

  • clinic-front-desk/frontend — React + MUI app
  • clinic-front-desk/backend — Node + Express + MongoDB app

So you have a monorepo-style layout: one repo, two deployable apps.

4. Backend (Node.js + Express + MongoDB) — Folders & Files

Root: backend/

4.1 Folder structure

backend/
├── src/
│   ├── config/           # env, db connection
│   ├── middleware/       # auth (JWT), error handler, validation
│   ├── models/           # Mongoose schemas (User, Doctor, Appointment, QueueEntry)
│   ├── controllers/      # request handlers (auth, doctors, appointments, queue)
│   ├── routes/           # route definitions (mount controllers)
│   ├── services/         # optional: business logic separate from controllers
│   ├── utils/            # helpers (e.g. JWT sign/verify, errors)
│   └── app.js or index.js
├── tests/                # optional: integration or unit tests
├── .env.example          # list required env vars (no secrets)
├── .env                  # actual secrets (never commit)
├── package.json
├── .gitignore
└── README.md

4.2 Files you must have (checklist)

  • Entry: src/app.js or src/index.js — create Express app, connect DB, mount routes, start server.
  • Config: src/config/db.js (or database.js) — MongoDB connection using MONGO_URI from env.
  • Env: .env (local only), .env.example with placeholders e.g. PORT=5000, MONGO_URI=, JWT_SECRET=.
  • Middleware:
    • src/middleware/auth.js — verify JWT, attach user to req.
    • src/middleware/errorHandler.js — central error handler.
    • Optional: src/middleware/validate.js — validate body/params (e.g. with express-validator).
  • Models (Mongoose):
    • src/models/User.js — front desk staff (email, hashed password, role).
    • src/models/Doctor.js — name, specialization, gender, location, availability.
    • src/models/Appointment.js — patient info, doctor ref, datetime, status (booked/completed/canceled).
    • src/models/QueueEntry.js — patient info, queue number, status (waiting/with_doctor/completed), optional priority.
  • Controllers:
    • src/controllers/authController.js — login, return JWT (and optionally "me").
    • src/controllers/doctorController.js — CRUD + search/filter.
    • src/controllers/appointmentController.js — list, create, update (reschedule), cancel; update status.
    • src/controllers/queueController.js — list queue, add patient, update status (and optional priority).
  • Routes:
    • src/routes/authRoutes.js — POST /login (and optionally GET /me).
    • src/routes/doctorRoutes.js — /doctors CRUD + GET with query params for search/filter.
    • src/routes/appointmentRoutes.js — /appointments CRUD + PATCH for status.
    • src/routes/queueRoutes.js — /queue GET, POST, PATCH.
    • src/routes/index.js — mount all routes under /api/*, apply auth middleware to protected routes.
  • Package: package.json — express, mongoose, jsonwebtoken, bcryptjs, dotenv, cors; dev: nodemon.
  • Git: .gitignore — node_modules, .env, logs.

You can add src/services/ later if you want to keep controllers thin.

5. Frontend (React + MUI) — Folders & Files

Root: frontend/

5.1 Folder structure

frontend/
├── public/
│   └── index.html
├── src/
│   ├── api/              # all API calls (axios/fetch to backend)
│   ├── components/       # reusable UI (buttons, modals, table wrappers)
│   │   ├── common/       # shared (e.g. Layout, Header, Sidebar)
│   │   └── ...

│   ├── context/          # auth context (current user, login/logout)
│   ├── hooks/            # optional: useAuth, useDoctors, etc.
│   ├── utils/            # helpers (e.g. format date, constants)
│   ├── App.jsx
│   └── main.jsx
├── .env.example          # VITE_API_URL or REACT_APP_API_URL
├── .env
├── package.json
├── .gitignore
└── README.md

5.2 Files you must have (checklist)

  • Entry: src/main.jsx — render React app; src/App.jsx — router and layout (MUI theme/provider here if you use one).
  • API layer:
    • src/api/axios.js (or client.js) — create axios instance with base URL from env, attach JWT from storage, handle 401 (e.g. logout).
    • src/api/auth.js — login(email, password).
    • src/api/doctors.js — getDoctors, getDoctor(id), create, update, delete, search(params).
    • src/api/appointments.js — list, create, update (reschedule), cancel, updateStatus.
    • src/api/queue.js — getQueue, addPatient, updateStatus (and optional priority).
  • Auth:
    • src/context/AuthContext.jsx — state: user, token, loading; actions: login, logout; read token from localStorage on init.
    • Optional: src/components/common/ProtectedRoute.jsx — redirect to login if not authenticated.
  • Pages (match PDF):
    • src/pages/LoginPage.jsx — form → call auth API → store token → redirect to dashboard/front-desk.
    • src/pages/FrontDeskPage.jsx or DashboardPage.jsx — landing after login; links/cards to Queue and Appointments.
    • src/pages/QueueManagementPage.jsx — list queue (number, status), add walk-in, update status (Waiting / With Doctor / Completed).
    • src/pages/AppointmentManagementPage.jsx — list doctors and slots, book/reschedule/cancel, show appointment status.
    • Optional: src/pages/DoctorsPage.jsx — list, add, edit, delete doctors; search/filter. You can merge "Doctors" into Appointment Management if you prefer fewer pages.
  • Components:
    • src/components/common/Layout.jsx — sidebar/nav + outlet for MUI layout.
    • Reusable pieces as needed: DataTable for queue and appointments, Modal/Dialog for forms, StatusChip for statuses.
  • Env: .env with VITE_API_URL=http://localhost:5000 (or REACT_APP_API_URL if using CRA); .env.example with same key, no real URL.
  • Package: package.json — react, react-dom, react-router-dom, @mui/material, @emotion/react, @emotion/styled, axios.
  • Git: .gitignore — node_modules, .env, build.

6. Step-by-Step Order of Execution (No Coding, Just Sequence)

  1. Setup
    • Create parent folder and frontend / backend subfolders.
    • Initialize both (npm init, then install dependencies; frontend: Vite+React or CRA).
    • Add .env.example and .gitignore in both; document required env vars in README.
  2. Backend first (so FE has an API to call)
    • Config: DB connection from env.
    • Models: User, Doctor, Appointment, QueueEntry.
    • Auth: login endpoint, JWT issue, auth middleware.
    • Routes: mount auth (public) and rest under /api/* with auth middleware.
    • Controllers: doctors CRUD + filter, appointments CRUD + status, queue list/add/update.
    • Test with Postman/Thunder Client (no UI yet).
  3. Frontend skeleton
    • API client (base URL, JWT attachment, 401 handling).
    • Auth context + login page; protected route wrapper.
    • Layout (sidebar/nav) and placeholder pages for Front Desk, Queue, Appointments (and Doctors if separate).
    • Wire login to backend and list pages to real endpoints (even simple lists).
  4. Frontend features (in order)
    • Doctors: list, search/filter, add/edit/delete (or integrate into appointments screen).
    • Appointments: view doctors/slots, book, reschedule, cancel, show status.
    • Queue: view queue, add walk-in, update status.
    • Polish: error messages, loading states, basic validation.
  5. Optional
    • Queue priority (e.g. "urgent" flag and sort).
    • Simple tests for critical API routes.
  6. Deploy
    • Backend: Deploy Express app to a Node host (e.g. Railway, Render, Fly.io). Set MONGO_URI (e.g. MongoDB Atlas) and JWT_SECRET in env. Expose one URL (e.g. https://your-api.railway.app).
    • Frontend: Build (npm run build), deploy dist/ to Vercel/Netlify. Set env to VITE_API_URL=https://your-api.railway.app for production. Ensure CORS on backend allows that origin.
    • Wrap: Update README with live FE and BE links, env vars, and how to run locally.

7. How to Use This Plan Day-to-Day

  • Product manager view: Use the four pillars (Auth, Doctors, Queue, Appointments) as milestones. Finish one pillar (backend + frontend) before moving to the next.
  • Engineering manager view: Use the file checklists above. Create empty files and folders first, then fill them in the order of the "Step-by-Step" section so you never block yourself (e.g. no FE work until auth and at least one API work).
  • Deployment: Treat deploy as a separate mini-phase: get BE running and DB connected, then point FE build to that API and fix CORS/env.

No code is included here—only structure, file names, and sequence. You can start by creating the folder trees and the listed files as stubs, then implement one pillar at a time.
